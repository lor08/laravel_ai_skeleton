# ADR-0006: Strategy for adopting the skeleton in an existing project

- **Status:** Accepted
- **Date:** YYYY-MM-DD
- **Deciders:** Project owner

## Context

`init.sh` создан под **новый** проект — он подменяет плейсхолдеры в чистом скелете и предполагает, что `composer install` ещё не был сделан, нет существующего legacy-кода.

В реальности часто нужно подружить скелет с **существующим** Laravel-проектом, у которого:
- Уже есть `composer.json`, `phpstan.neon`, тесты, CI
- Есть legacy-код, не соответствующий правилам (final, strict_types, helpers vs facades)
- Своя ветка по умолчанию, свой ticket-prefix, свой стиль коммитов

Запустить `init.sh` поверх существующего проекта — опасно: он перепишет файлы, плейсхолдеры подставит криво, sed может зацепить чужие строки.

## Decision

Существующие проекты **не используют `init.sh`**. Вместо этого — документ **`ADOPTION.md`** с пошаговой инструкцией по 3 уровням:

- **L1 — Knowledge only:** копируем `.ai/`, `AGENTS.md`, `CLAUDE.md`. Код не меняется. Агенты получают правила.
- **L2 — Soft enforcement:** добавляем `.claude/` с командами и хуками. Жёсткие хуки переводим в soft-режим, чтобы legacy не блокировал работу.
- **L3 — Full CI gate:** Pest arch + PHPStan baseline + ECS, новый код блокируется при нарушении, старый прощается через `ignoring()` / baseline.

Стратегия rollout:
- Маленькие проекты — за день дотащить до L3
- Средние — L2 несколько дней, потом L3 с baseline
- Большие/legacy — L2 неделями, L3 поэтапно по модулям

## Alternatives Considered

### Option A — `init.sh --adopt` режим

- ✅ Один скрипт на оба сценария
- ❌ Слишком много веток в скрипте: что копировать, что мерджить, что не трогать
- ❌ Высокий риск повреждения существующих файлов (composer.json, phpstan.neon)

### Option B — Жёсткое требование «новый проект только»

- ✅ Простота скелета
- ❌ Не работает на больших командах с легаси-кодом — никто не будет переписывать с нуля

### Option C (Chosen) — `init.sh` для новых, `ADOPTION.md` для существующих

- ✅ Каждый сценарий — со своим инструментом
- ✅ ADOPTION.md по уровням снижает риск регрессии
- ✅ PHPStan baseline и arch `ignoring()` — стандартные механизмы для legacy
- ❌ Нужно поддерживать два пути

## Consequences

### Positive

- Скелет можно внедрять в любой Laravel-проект, не только в чистый
- Постепенное ужесточение через 3 уровня — командам комфортно
- Baseline для PHPStan и ignoring для arch — стандартный инструментарий, всем знаком

### Negative

- Дополнительный документ для поддержки
- При обновлении правил скелета — нужны заметки про L2/L3 миграции

### Mitigation

- ADOPTION.md содержит чек-лист после adoption — можно прогнать после каждого обновления скелета
- Baseline должен **уменьшаться** со временем, не расти — это правило проекта, не временная мера
