# Documentation Rules

Документация — **часть Definition of Done**. Без её обновления задача не считается завершённой.

## Definition of Done — чек-лист документации

После реализации, перед отчётом, пройди по списку:

| Триггер | Куда обновить |
|---|---|
| Появился новый термин домена | `.ai/project/glossary.md` |
| Новая бизнес-логика модуля / нюанс | `.ai/project/domain/{module}.md` *(создать, если нет)* |
| Новая внешняя интеграция / API | `.ai/project/integrations.md` |
| Новый cron, queue, кэш, очередь, особенность инфры | `.ai/project/operations.md` |
| Найден подвох, который не понять из кода | `.ai/project/gotchas.md` |
| Принято архитектурное решение | `.ai/adr/{NNNN}-{title}.md` *(новый)* |
| Изменилась схема БД (новые таблицы / связи / инварианты) | `.ai/project/data-model.md` |
| Описана процедура восстановления / переиндексации / миграции | `.ai/project/runbook.md` |
| Стандарт кода или паттерн поменялся | `.ai/rules/...` |
| Проект поменял фокус, стек, ключевую фичу | `.ai/project/overview.md` |

## Что НЕ писать в проектные доки

- Состояние текущей задачи → `.ai/tasks/{ticket}-decisions.md`
- Заметки между сессиями (короткие) → `.ai/memory.md`
- Секреты, токены, ключи → нигде в репозитории

## Когда создавать ADR

Создаём ADR, если решение:

- **Архитектурное** (паттерн, граница, технология)
- **Долгоиграющее** (откатить будет дорого)
- **Не очевидно** из кода (есть альтернативы, и не понятно, почему выбрали эту)

Когда **не** создаём:

- Решение прозрачно из кода (например, «использовали array_map» — не ADR)
- Это локальная тактика, а не стратегия
- Это бизнес-правило (оно идёт в `.ai/project/domain/`, а не в ADR)

## Шаблон ADR

См. `.ai/adr/0000-template.md`.

## Когда обновлять `.ai/memory.md`

`.ai/memory.md` — короткие заметки между сессиями. Туда:

- Текущая ветка / тикет
- Особенности окружения (если временно отличается от operations.md)
- Незакрытые «висяки», к которым надо вернуться
- Наблюдения, которые ещё не дозрели до правила или ADR

Раз в N сессий перечитай и почисти / промотируй в правила/ADR.

## Правило для агента

В **каждом** `/task` и `/quick` workflow секция «Документация» — обязательная.
Если ничего обновлять не пришлось — явно зафиксируй это в отчёте: «Документация: изменений не требуется».

## Hook напоминания

`.claude/hooks/doc-reminder.sh` (Stop hook) проверяет, что если в сессии были Edit/Write на чувствительные пути (`app/Modules/`, `database/migrations/`, `composer.json` deps, `config/`), а в `.ai/project/` или `.ai/adr/` ничего не менялось — напомнит в stderr.

Это **мягкое** напоминание (warn, не block). Главный заслон — workflow rule выше.

## Команда `/update-docs`

Если за сессию накопилось много изменений и хочется явно выгрузить их в доку — `/update-docs` запустит агента: посмотреть `git diff <MAIN-BRANCH>...HEAD`, сопоставить с матрицей выше, предложить конкретные правки в `.ai/project/`.
