# Workflow

Базовый принцип: **анализ → план → согласование → код → проверки → документация → отчёт**.
Не кодим до согласия пользователя на план.

## Два режима

### `/quick` — лёгкий (мелкие правки)

Когда: правка одного-двух файлов, понятный фикс, тривиальная фича.

1. Прочитать релевантные правила (`.ai/rules/...`) и затронутые файлы
2. Сформулировать **план в 5 пунктов** (что меняем, где, в каком порядке)
3. Показать пользователю → дождаться `ок`
4. Реализовать
5. Запустить `composer style && composer analyse && composer arch && composer test`
6. Обновить `.ai/project/...` если нужно (см. `docs.md`)
7. Краткий отчёт

### `/task` — тяжёлый (крупные задачи)

Когда: новый модуль, рефакторинг нескольких слоёв, фича, затрагивающая фронт+бэк.

1. **Подготовка**
   - Прочитать `AGENTS.md`, `.ai/STARTUP.md`, релевантные `.ai/rules/*` и `.ai/project/*`
   - Получить тикет / описание задачи (`$ARGUMENTS`)
   - Спросить пользователя: «Делаем `decisions.md` или сразу к плану?» *(decisions нужен, если задача крупная и решений много)*
   - Если decisions: создать `.ai/tasks/<TICKET-PREFIX>-{NNNN}-decisions.md`
2. **Архитектурное интервью** — блок за блоком, с вариантами A/B/C:
   - Блок 1 — Данные (схема БД, миграции, модели, енумы)
   - Блок 2 — Бэкенд API (контроллеры, маршруты, FormRequests, сервисы, репозитории, ресурсы, jobs, events)
   - Блок 3 — Фронтенд (компоненты, API-клиент, store, UX)
   - Блок 4 — Интеграции (граница модулей, bridges, shared-код)
   - Каждое решение записывается в `decisions.md` со статусом `✅ ПРИНЯТО`
3. **Реализация в порядке зависимостей** (см. ниже)
4. **Проверки** — `composer all`
5. **Документация** — обязательно (см. `docs.md`)
6. **Финальный отчёт**

## Порядок зависимостей при реализации

1. Миграции (схема первична)
2. Models / Enums / DTOs
3. Contracts (интерфейсы)
4. Repositories
5. Services (бизнес-логика)
6. FormRequests + Resources
7. Controllers / Console commands
8. Jobs / Events / Listeners
9. Frontend (компоненты, store, API-клиент)
10. Lang / Routes / Sidebar
11. Тесты (unit + feature + arch)
12. Cleanup (удалить мёртвый код, MVP-заглушки)

После каждого PHP-файла — `<RUN-CMD> ecs check --fix path/to/File.php`.

## decisions.md шаблон

```markdown
# <TICKET-PREFIX>-NNNN — Название задачи

> **Читать перед каждой сессией.**

## Описание
Краткое описание задачи.

## Статус

| Блок | Статус |
|---|---|
| Данные / Миграции | ⏳ Pending |
| Бэкенд API | ⏳ Pending |
| Фронтенд | ⏳ Pending |
| Интеграции | ⏳ Pending |

---

## Архитектурные решения

### 1 — {название}
{описание принятого решения}
✅ **ПРИНЯТО**

---

## Чек-лист реализации

### Backend
- [ ] ...

### Frontend
- [ ] ...

### Quality
- [ ] composer style
- [ ] composer analyse
- [ ] composer arch
- [ ] composer test
- [ ] Браузер
- [ ] Документация (.ai/project/, ADR при необходимости)
```

## Когда останавливаться и спрашивать

- Решение в `decisions.md` неоднозначно
- Файл/код удивил (неожиданные зависимости, что-то ломается)
- Появился конфликт с правилами проекта
- Найдено решение, которого нет в правилах — предложить добавить ADR или правило

## Финальный отчёт

```markdown
## Реализовано

### Создано
- `path/to/File.php` — что делает

### Изменено
- `path/to/File.php` — что изменено

### Удалено
- `path/to/Dead.php`

### Проверки
- [x] composer style: OK
- [x] composer analyse: OK
- [x] composer arch: OK
- [x] composer test: X passed
- [ ] Браузер: проверено / не проверено (причина)

### Документация
- [x] `.ai/project/glossary.md` — добавлены термины X, Y
- [x] `.ai/adr/0007-...md` — создано
- [ ] `.ai/project/operations.md` — не требовалось
```
